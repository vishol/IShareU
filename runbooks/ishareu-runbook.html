Application Description

IShareU is a web-based SaaS file-sharing application. Registered users can upload one or more files, define expiry and max-download rules, and share generated public links. Recipients access a landing download page and are then redirected to a short-lived signed URL generated from Supabase Storage.

Core Processing Capabilities

  • User registration, login, and role-based admin access through Spring Security.
  • Multi-file upload with size checks, allow-list validation, and per-IP upload rate limiting.
  • Public link-based file access with expiry enforcement, max-download enforcement, and per-IP download rate limiting.
  • File metadata persistence in PostgreSQL through Spring Data JPA.
  • Cloud object upload and signed download URL generation through Supabase Storage REST APIs.
  • Admin dashboards for user and file searches, pagination, and record deletion.

Application Overview

The application follows a conventional Spring MVC layout. Browser requests hit Thymeleaf-backed controllers, business logic is handled in service classes, file metadata is stored in PostgreSQL via JPA repositories, and file binaries live in a Supabase bucket. Security rules allow public access to home, health, and download pages while requiring authentication for uploads and admin privileges for `/admin/**` endpoints.

Primary runtime flow: browser -> controller -> service -> repository and Supabase Storage -> Thymeleaf view or redirect.

Processing Steps

  1. User signs up or logs in through Spring Security form login.
  2. Authenticated user opens `/file/upload` and submits one or more files.
  3. `FileController` enforces per-IP bucket limits, validates file size and extension/content type against the configured allow-list, and resolves the current user.
  4. `SupabaseStorageService` uploads each accepted file to the configured Supabase bucket and returns a generated storage filename.
  5. `FileController` builds a `FileRecord` with original filename, storage key, generated unique link, expiry, ownership, and max-download metadata, then persists it through `FileService`.
  6. The upload-success page displays generated public links.
  7. When a recipient opens the link, the app verifies that the record exists, is active, is not expired, and has not exceeded the download limit.
  8. The app generates a one-hour signed URL from Supabase Storage, increments the download count, and redirects the recipient to the object store.
  9. A scheduled cleanup job runs hourly and deactivates expired or download-limit-reached records.
  10. Admins can review aggregate stats, search users/files, and delete records from dedicated admin pages.

Operational Notes

The current repository contains live-looking database and Supabase credentials in src/main/resources/application.yml. Treat that as an immediate security risk and rotate those credentials before relying on this deployment.
